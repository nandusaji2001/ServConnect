1. CONTENT MODERATION MODULE
================================================================================

File: content_moderation_api.py
Training Script: train_model.py
Purpose: Detect harmful/toxic content in user posts and comments

ALGORITHMS USED:
----------------

1.1 LOGISTIC REGRESSION (Primary Classifier)
   - Type: Supervised Learning - Binary Classification
   - Library: scikit-learn (sklearn.linear_model.LogisticRegression)
   - Configuration:
     * Solver: lbfgs (Limited-memory BFGS)
     * Regularization: C=1.0 (inverse of regularization strength)
     * Max iterations: 1000
     * Class weight: balanced (handles imbalanced data)
     * Multi-core: n_jobs=-1 (uses all CPU cores)
   
   - Why Logistic Regression:
     * Fast training and prediction
     * Interpretable coefficients
     * Works well with TF-IDF features
     * Good for binary classification
     * Low memory footprint
   
   - Performance: ~95% accuracy on toxic comment detection

1.2 TF-IDF VECTORIZATION (Feature Extraction)
   - Type: Natural Language Processing - Text Vectorization
   - Library: scikit-learn (sklearn.feature_extraction.text.TfidfVectorizer)
   - Configuration:
     * Max features: 10,000 (vocabulary size)
     * N-gram range: (1, 2) (unigrams and bigrams)
     * Min document frequency: 2
     * Max document frequency: 0.95
     * Stop words: English
   
   - What is TF-IDF:
     * TF (Term Frequency): How often a word appears in a document
     * IDF (Inverse Document Frequency): How rare/common a word is across all documents
     * Formula: TF-IDF = TF × log(N/DF)
       where N = total documents, DF = documents containing the term
   
   - Why TF-IDF:
     * Captures word importance better than simple word counts
     * Reduces weight of common words (like "the", "is")
     * Increases weight of distinctive words
     * Creates sparse matrix (memory efficient)

1.3 TEXT PREPROCESSING
   - Techniques Used:
     * Lowercasing: Convert all text to lowercase
     * URL removal: Remove http/https links
     * HTML tag removal: Strip HTML tags
     * Special character removal: Keep only letters and spaces
     * Whitespace normalization: Remove extra spaces
   
   - Regular Expressions (Regex):
     * Pattern matching for text cleaning
     * URL pattern: r'http\S+|www\S+|https\S+'
     * HTML pattern: r'<.*?>'
     * Special chars: r'[^a-zA-Z\s]'

DATASET:
--------
- Source: Kaggle Toxic Comment Classification Challenge
- File: Data/train.csv
- Size: ~160,000 comments
- Labels: toxic, severe_toxic, obscene, threat, insult, identity_hate
- Binary target: is_harmful (1 if any toxic category is 1, else 0)

OUTPUT:
-------
- Prediction: is_harmful (True/False)
- Confidence score: 0.0 to 1.0
- Threshold: 0.5 (configurable)

2. ELDER WELLNESS PREDICTION MODULE
================================================================================

File: elder_wellness_api.py
Training Script: train_wellness_model.py
Purpose: Predict diet recommendations, diet plans, and heart risk for elderly

ALGORITHMS USED:
----------------

2.1 RANDOM FOREST CLASSIFIER (Primary Algorithm)
   - Type: Supervised Learning - Multi-class Classification
   - Library: scikit-learn (sklearn.ensemble.RandomForestClassifier)
   - Configuration:
     * Number of trees: 100 (n_estimators=100)
     * Max depth: 10 (prevents overfitting)
     * Random state: 42 (reproducibility)
     * Multi-core: n_jobs=-1 (parallel processing)
   
   - How Random Forest Works:
     * Ensemble method: Combines multiple decision trees
     * Bootstrap aggregating (bagging): Each tree trained on random subset
     * Feature randomness: Each split considers random subset of features
     * Voting: Final prediction is majority vote from all trees
   
   - Why Random Forest:
     * Handles non-linear relationships well
     * Robust to outliers and noise
     * Provides feature importance
     * Reduces overfitting compared to single decision tree
     * Works well with mixed data types (numeric + categorical)
     * No need for feature scaling
   
   - Three Separate Models Trained:
     a) Diet Recommendation Model
        - Predicts: Specific dietary advice based on health metrics
        - Classes: Multiple recommendation types
        - Accuracy: ~85-90%
     
     b) Diet Plan Model
        - Predicts: vegetarian, vegan, keto, high_protein
        - Classes: 4 diet types
        - Accuracy: ~88-92%
     
     c) Heart Risk Model
        - Predicts: low, moderate, high
        - Classes: 3 risk levels
        - Accuracy: ~90-95%

2.2 LABEL ENCODING (Categorical Feature Encoding)
   - Type: Data Preprocessing
   - Library: scikit-learn (sklearn.preprocessing.LabelEncoder)
   - Purpose: Convert categorical variables to numeric
   
   - Encoded Features:
     * Gender: male → 0, female → 1
     * Sleep quality: poor → 0, fair → 1, good → 2
     * Physical activity: sedentary → 0, light → 1, moderate → 2, active → 3
   
   - Why Label Encoding:
     * Machine learning algorithms require numeric input
     * Preserves ordinal relationships (for ordered categories)
     * Memory efficient
     * Fast encoding/decoding

2.3 FEATURE ENGINEERING
   - Input Features (13 total):
     * age: Elder's age in years
     * gender: Male/Female
     * bmi: Body Mass Index
     * systolic_bp: Systolic blood pressure
     * diastolic_bp: Diastolic blood pressure
     * cholesterol: Total cholesterol level
     * triglycerides: Triglyceride level
     * family_history_t2d: Family history of Type 2 Diabetes (0/1)
     * family_history_cvd: Family history of cardiovascular disease (0/1)
     * sleep_hours: Hours of sleep per night
     * sleep_quality: poor/fair/good
     * stress_level: Stress level (1-10)
     * physical_activity_level: sedentary/light/moderate/active

DATASET:
--------
- File: Data/elder_diet_ml_ready_dataset_v3.csv
- Size: ~1000+ elder health records
- Features: 13 input features
- Targets: 3 (diet_recommendation, diet_plan, heart_risk)

OUTPUT:
-------
- Diet recommendation: Personalized dietary advice
- Diet plan: Suggested diet type (vegetarian/vegan/keto/high_protein)
- Heart risk: Risk level (low/moderate/high)
- Detailed exercise recommendations based on risk
- 7-day meal plans with recipes

3. ID VERIFICATION MODULE
================================================================================

File: id_verification_api.py
Purpose: Verify identity by extracting name from ID card images using OCR

ALGORITHMS USED:
----------------

3.1 EASYOCR (Optical Character Recognition)
   - Type: Deep Learning - Computer Vision
   - Library: EasyOCR (easyocr.Reader)
   - Configuration:
     * Languages: English ('en')
     * GPU: False (uses CPU for compatibility)
   
   - How EasyOCR Works:
     * Detection Network: Locates text regions in image
       - Uses CRAFT (Character Region Awareness For Text detection)
       - Detects text at character level
       - Groups characters into words
     
     * Recognition Network: Reads detected text
       - Uses CRNN (Convolutional Recurrent Neural Network)
       - CNN extracts visual features
       - RNN (LSTM) sequences features into text
       - CTC (Connectionist Temporal Classification) decodes output
   
   - Why EasyOCR:
     * Pre-trained on large datasets
     * Supports multiple languages
     * Works with various ID card formats
     * Handles rotated and skewed text
     * Good accuracy on Indian ID cards (Aadhar, PAN, Voter ID)
     * Easy to use API

3.2 FUZZY STRING MATCHING (Name Comparison)
   - Type: String Similarity Algorithm
   - Library: Python difflib (SequenceMatcher)
   - Algorithm: Gestalt Pattern Matching
   
   - How it Works:
     * Compares two strings character by character
     * Finds longest common subsequences
     * Calculates similarity ratio: 2*M / T
       where M = matching characters, T = total characters
     * Returns score from 0.0 (no match) to 1.0 (perfect match)
   
   - Why Fuzzy Matching:
     * Handles typos and OCR errors
     * Accounts for spelling variations
     * Case-insensitive comparison
     * Tolerates minor differences
   
   - Threshold: 0.75 (75% similarity required for auto-approval)

3.3 TEXT PREPROCESSING & NAME EXTRACTION
   - Techniques:
     * Lowercasing: Normalize case
     * Whitespace normalization: Remove extra spaces
     * Special character removal: Keep only letters
   
   - Name Extraction Heuristics:
     * Pattern matching: Look for "Name:", "Father:", etc.
     * Standalone names: Lines with only alphabetic characters
     * Position-based: Name usually after indicator keywords
     * Confidence filtering: Use OCR confidence scores
   
   - Skip Words (Noise Filtering):
     * Common labels: "government", "india", "identity", "card"
     * Field names: "address", "date", "birth", "dob"
     * Gender markers: "male", "female"

3.4 IMAGE PREPROCESSING
   - Techniques:
     * Format conversion: RGBA → RGB (for JPEG compatibility)
     * White background: For transparent images
     * Quality preservation: Save at 95% JPEG quality
     * Temporary file handling: Clean up after processing

DATASET:
--------
- No training dataset (uses pre-trained EasyOCR models)
- Supports: Aadhar Card, PAN Card, Voter ID, Driving License

OUTPUT:
-------
- verified: True/False (auto-approval decision)
- similarity_score: 0.0 to 1.0
- extracted_names: List of potential names found
- best_match: Best matching name with confidence
- message: Verification result message

4. DEPRESSION PREDICTION & WELLNESS MODULE
================================================================================

File: depression_prediction_api.py
Training Script: train_depression_model.py
Purpose: Predict depression status and generate personalized wellness tasks

ALGORITHMS USED:
----------------

4.1 XGBOOST CLASSIFIER (Primary Algorithm)
   - Type: Supervised Learning - Binary Classification
   - Library: XGBoost (xgboost.XGBClassifier)
   - Configuration:
     * Number of trees: 200 (n_estimators=200)
     * Max depth: 6 (tree depth)
     * Learning rate: 0.1 (eta)
     * Subsample: 0.8 (80% of data per tree)
     * Column sample: 0.8 (80% of features per tree)
     * Scale pos weight: Auto-calculated for imbalanced classes
     * Eval metric: logloss (log loss for binary classification)
   
   - How XGBoost Works:
     * Gradient Boosting: Sequential ensemble method
     * Each tree corrects errors of previous trees
     * Uses gradient descent to minimize loss function
     * Regularization: L1 (Lasso) and L2 (Ridge) to prevent overfitting
     * Tree pruning: Removes splits that don't improve performance
     * Parallel processing: Fast training on multiple cores
   
   - Why XGBoost:
     * State-of-the-art performance on tabular data
     * Handles missing values automatically
     * Built-in regularization prevents overfitting
     * Feature importance ranking
     * Handles imbalanced classes well
     * Fast training and prediction
     * Better than Random Forest for complex patterns
   
   - Performance: ~88-92% accuracy on depression prediction

4.2 LABEL ENCODING (Categorical Features)
   - Encoded Features:
     * Gender: Male/Female
     * Sleep Duration: <5 hours, 5-6 hours, 6-7 hours, 7-8 hours, >8 hours
     * Dietary Habits: Unhealthy, Moderate, Healthy
     * Family History: Yes/No
     * Suicidal Thoughts: Yes/No
     * User Type: Student/Working Professional
   
   - Why Label Encoding:
     * XGBoost requires numeric input
     * Preserves ordinal relationships
     * Memory efficient

4.3 FEATURE ENGINEERING
   - Common Features (8):
     * Gender, Age, Sleep Duration, Dietary Habits
     * Work/Study Hours, Financial Stress
     * Family History of Mental Illness
     * Suicidal Thoughts History
   
   - Student-Specific Features (3):
     * Academic Pressure (1-5 scale)
     * CGPA (0-10 scale)
     * Study Satisfaction (1-5 scale)
   
   - Working Professional Features (2):
     * Work Pressure (1-5 scale)
     * Job Satisfaction (1-5 scale)
   
   - Feature Handling:
     * Students: Work features set to neutral (3)
     * Working Professionals: Student features set to neutral
     * Missing values: Filled with median or neutral values

4.4 CLASS IMBALANCE HANDLING
   - Technique: scale_pos_weight parameter
   - Formula: scale_pos_weight = count(negative) / count(positive)
   - Effect: Penalizes misclassification of minority class more
   - Why: Depression cases are typically minority class

4.5 WELLNESS TASK GENERATION (Rule-Based System)
   - Algorithm: Personalized recommendation engine
   - Inputs:
     * Depression probability (0.0 to 1.0)
     * User location (for place recommendations)
     * Severity level (low/medium/high)
   
   - Task Categories (6):
     * Outdoor activities
     * Social connections
     * Mindfulness & relaxation
     * Physical wellness
     * Creative expression
     * Self-care & routine
   
   - Task Selection Logic:
     * High severity (>0.7): 4 tasks per day
     * Medium/Low severity: 3 tasks per day
     * Place visits: Only on Tue/Thu/Sat
     * Random selection for variety
     * Daily affirmations
   
   - For Non-Depressed Users:
     * Maintenance wellness plan
     * Lighter tasks (2-3 per day)
     * Focus on prevention
     * No place visits (simpler routine)

DATASET:
--------
- File: Data/final_depression_dataset_1.csv
- Size: ~2000+ records
- Features: 13 total (8 common + 3 student + 2 working)
- Target: Depression (Yes/No)
- User Types: Students and Working Professionals

OUTPUT:
-------
- Prediction: isDepressed (True/False)
- Confidence: 0.0 to 1.0
- Severity level: low/medium/high
- 7-day wellness plan with personalized tasks
- Daily affirmations
- Recommendations based on severity

5. LOST & FOUND MATCHING MODULE (OLD - SBERT)
================================================================================

File: item_matching_api.py
Purpose: Match lost and found items using text similarity

ALGORITHMS USED:
----------------

5.1 SENTENCE-BERT (S-BERT) - Text Embeddings
   - Type: Deep Learning - Natural Language Processing
   - Library: sentence-transformers
   - Model: all-MiniLM-L6-v2
   
   - How S-BERT Works:
     * Based on BERT (Bidirectional Encoder Representations from Transformers)
     * Modified for sentence embeddings
     * Uses Siamese network architecture
     * Produces fixed-size embeddings (384 dimensions)
   
   - Architecture:
     * Input: Text sentence
     * Tokenization: WordPiece tokenizer
     * BERT Encoder: 6 layers, 384 hidden units
     * Pooling: Mean pooling of token embeddings
     * Output: 384-dimensional vector
   
   - Why S-BERT:
     * Captures semantic meaning (not just keywords)
     * "Black wallet" and "Dark purse" are similar
     * Pre-trained on large corpus
     * Fast inference (compared to BERT)
     * Good for similarity tasks
   
   - Model Details:
     * Parameters: ~22 million
     * Training: Trained on 1 billion sentence pairs
     * Speed: ~2000 sentences/second on CPU

5.2 COSINE SIMILARITY (Similarity Metric)
   - Type: Vector Similarity Measure
   - Library: scikit-learn (sklearn.metrics.pairwise.cosine_similarity)
   
   - Formula:
     similarity = (A · B) / (||A|| × ||B||)
     where A and B are embedding vectors
   
   - Range: -1 to 1 (typically 0 to 1 for text)
     * 1.0 = Identical
     * 0.5 = Moderately similar
     * 0.0 = Completely different
   
   - Why Cosine Similarity:
     * Measures angle between vectors (not magnitude)
     * Works well with high-dimensional embeddings
     * Normalized (0-1 range)
     * Fast computation

5.3 CATEGORY BOOST (Heuristic Enhancement)
   - Type: Rule-based scoring adjustment
   - Logic: If categories match, add 0.15 to similarity
   - Example:
     * Text similarity: 0.70
     * Both items are "Wallet"
     * Final score: min(1.0, 0.70 + 0.15) = 0.85
   
   - Why Category Boost:
     * Category is strong signal for matching
     * Improves precision
     * Reduces false positives

5.4 TEXT PREPROCESSING
   - Techniques:
     * Combine: title + category + description + location
     * Lowercase: Normalize case
     * Whitespace: Clean extra spaces
   
   - Example:
     Input: {"title": "Black Wallet", "category": "Wallet", 
             "description": "Lost near park", "location": "Central Park"}
     Output: "Black Wallet Category: Wallet Lost near park Location: Central Park"

DATASET:
--------
- No training dataset (uses pre-trained S-BERT model)
- Real-time matching on user-submitted items

OUTPUT:
-------
- Matches: List of similar items
- Similarity score: 0.0 to 1.0
- Match percentage: 0% to 100%
- Ranked by similarity (descending)

LIMITATIONS:
------------
- Text-only (no image support)
- No user trust scoring
- No fraud detection
- ~70% accuracy

6. LOST & FOUND MATCHING MODULE (NEW - MULTIMODAL + GNN)
================================================================================

Files: multimodal_item_matching_api.py, clip_service.py, gnn_service.py
Purpose: Advanced matching using images, text, and trust scoring

ALGORITHMS USED:
----------------

6.1 CLIP (Contrastive Language-Image Pre-training)
   - Type: Deep Learning - Multimodal AI
   - Library: HuggingFace Transformers
   - Model: openai/clip-vit-base-patch32
   
   - How CLIP Works:
     * Dual Encoder Architecture:
       a) Image Encoder: Vision Transformer (ViT)
          - Splits image into 32×32 patches
          - Processes patches through transformer
          - Outputs 512-dim image embedding
       
       b) Text Encoder: Transformer
          - Tokenizes text (max 77 tokens)
          - Processes through transformer layers
          - Outputs 512-dim text embedding
     
     * Contrastive Learning:
       - Trained on 400 million (image, text) pairs
       - Learns to match images with correct captions
       - Maximizes similarity for correct pairs
       - Minimizes similarity for incorrect pairs
   
   - Architecture Details:
     * Image: ViT-B/32 (Vision Transformer Base, 32×32 patches)
     * Layers: 12 transformer layers
     * Hidden size: 512 dimensions
     * Attention heads: 8
     * Parameters: ~151 million
   
   - Why CLIP:
     * Understands both images and text
     * Zero-shot learning capability
     * Cross-modal matching (image ↔ text)
     * Pre-trained on massive dataset
     * State-of-the-art multimodal understanding
   
   - Capabilities:
     * Image → Image similarity
     * Text → Text similarity
     * Image → Text similarity (cross-modal)
     * All embeddings in same 512-dim space

6.2 GRAPH NEURAL NETWORK (GNN) - Trust Scoring
   - Type: Deep Learning - Graph Learning
   - Library: PyTorch Geometric
   - Architecture: GraphSAGE (Graph Sample and Aggregate)
   
   - How GraphSAGE Works:
     * Graph Structure:
       - Nodes: Users and Items
       - Edges: User→Item (posted, claimed, viewed)
     
     * Message Passing:
       1. Sample neighbors for each node
       2. Aggregate neighbor features (mean pooling)
       3. Concatenate with node's own features
       4. Apply neural network transformation
       5. Repeat for multiple layers
     
     * Layer 1: Input features → 64 dimensions
       - Aggregation: Mean of neighbor features
       - Activation: ReLU
       - Dropout: 0.3 (prevents overfitting)
     
     * Layer 2: 64 dimensions → 32 dimensions
       - Aggregation: Mean of neighbor features
       - Output: Node embeddings
     
     * Trust Score: Sigmoid(embedding_norm)
       - Converts embedding to 0-1 score
   
   - Node Features:
     * Users: [posts_count, reports_count, account_age_days, is_user=1]
     * Items: [claims_count, verified, age_days, is_item=0]
   
   - Why GraphSAGE:
     * Captures network structure
     * Learns from user-item interactions
     * Inductive learning (works on new nodes)
     * Scalable to large graphs
     * Better than simple heuristics
   
   - Alternative: GAT (Graph Attention Network)
     * Uses attention mechanism
     * Learns importance of neighbors
     * More expressive but slower
     * Configurable via use_gat=True

6.3 MULTIMODAL SIMILARITY SCORING
   - Algorithm: Weighted combination of similarities
   
   - Formula:
     similarity = 0.4 × image_sim 
                + 0.4 × text_sim 
                + 0.2 × cross_modal_sim
   
   - Components:
     a) Image Similarity:
        - CLIP image embeddings
        - Cosine similarity
        - Compares item photos
     
     b) Text Similarity:
        - CLIP text embeddings
        - Cosine similarity
        - Compares descriptions
     
     c) Cross-Modal Similarity:
        - Image1 ↔ Text2 similarity
        - Text1 ↔ Image2 similarity
        - Average of both
   
   - Why This Weighting:
     * Images and text equally important (0.4 each)
     * Cross-modal provides additional signal (0.2)
     * Configurable weights

6.4 HYBRID SCORING (Similarity + Trust)
   - Algorithm: Weighted combination
   
   - Formula:
     final_score = α × similarity_score + β × trust_score
   
   - Default Weights:
     * α = 0.7 (70% similarity)
     * β = 0.3 (30% trust)
   
   - Trust Score Calculation:
     trust_score = (user_trust + item_trust) / 2
   
   - Why Hybrid Scoring:
     * Similarity: Matches item characteristics
     * Trust: Prevents fraud and spam
     * Configurable α/β for different priorities
     * Balances accuracy and safety

6.5 CATEGORY BOOST (Enhanced)
   - Same as old system: +0.1 if categories match
   - Applied after multimodal similarity
   - Improves precision

COMPARISON WITH OLD SYSTEM:
---------------------------

Feature                  | Old (SBERT)    | New (CLIP+GNN)
-------------------------|----------------|----------------
Text matching            | ✓ (S-BERT)     | ✓ (CLIP)
Image matching           | ✗              | ✓ (CLIP)
Cross-modal              | ✗              | ✓ (CLIP)
Trust scoring            | ✗              | ✓ (GNN)
Fraud detection          | ✗              | ✓ (GNN)
Embedding dimension      | 384            | 512
Accuracy                 | ~70%           | ~85-90%
Model size               | ~90MB          | ~700MB
Inference time           | ~100ms         | ~1-2 sec

DATASET:
--------
- CLIP: Pre-trained on 400M image-text pairs
- GNN: Trained on user-item interaction graph
- No additional training required

OUTPUT:
-------
- Matches: Ranked list of similar items
- Similarity score: 0.0 to 1.0
- Trust score: 0.0 to 1.0
- Final score: 0.0 to 1.0
- Detailed breakdown:
  * Image similarity
  * Text similarity
  * Cross-modal similarity
  * User trust
  * Item trust

7.1 TEXT TOXICITY DETECTION (Dual-Mode)

A) LEGACY MODE: TF-IDF + LOGISTIC REGRESSION
   - Same as Content Moderation Module (Section 1)
   - Fast, lightweight, backward compatible
   - Accuracy: ~95%

B) TRANSFORMER MODE: BERT/DistilBERT
   - Type: Deep Learning - Transformer Architecture
   - Library: HuggingFace Transformers
   - Model: distilbert-base-uncased-finetuned-sst-2-english (default)
   - Alternative: unitary/toxic-bert, martin-ha/toxic-comment-model
   
   - How BERT Works:
     * Bidirectional Transformer Encoder
     * Self-attention mechanism
     * Processes entire sentence context
     * Pre-trained on massive text corpus
     * Fine-tuned for toxicity classification
   
   - Architecture:
     * Input: Text (max 512 tokens)
     * Tokenization: WordPiece tokenizer
     * Encoder: 6 layers (DistilBERT) or 12 layers (BERT)
     * Hidden size: 768 dimensions
     * Classification head: Linear layer → 2 classes
     * Output: Toxicity probability (0-1)
   
   - Why BERT:
     * Understands context better than TF-IDF
     * Captures semantic meaning
     * Detects subtle toxicity
     * State-of-the-art NLP performance
     * Transfer learning from pre-training
   
   - Performance:
     * Accuracy: ~97-98% (better than legacy)
     * Speed: ~50ms per text (slower than legacy)
     * Model size: ~250MB (larger than legacy)

7.2 IMAGE CONTENT ANALYSIS (CLIP)
   - Type: Deep Learning - Multimodal AI
   - Library: HuggingFace Transformers
   - Model: openai/clip-vit-base-patch32
   
   - Same architecture as Lost & Found CLIP (Section 6.1)
   
   - Harmful Content Detection:
     * Harmful keywords: ["violence", "hate speech", "offensive content",
                          "explicit content", "harassment", "bullying",
                          "threatening", "disturbing imagery"]
     * Safe keywords: ["friendly", "positive", "educational",
                       "informative", "helpful"]
   
   - Scoring Algorithm:
     1. Encode image → 512-dim embedding
     2. Encode each harmful keyword → 512-dim embeddings
     3. Compute cosine similarity: image ↔ each harmful keyword
     4. Average harmful similarities → avg_harmful
     5. Encode each safe keyword → 512-dim embeddings
     6. Compute cosine similarity: image ↔ each safe keyword
     7. Average safe similarities → avg_safe
     8. Risk score = (avg_harmful - avg_safe + 1) / 2
     9. Normalize to [0, 1]
   
   - Image-Text Consistency:
     * Computes similarity between image and caption
     * Detects mismatched content (e.g., violent image with innocent caption)
     * Helps identify deceptive posts
   
   - Why CLIP for Moderation:
     * Understands visual content semantically
     * No need for explicit training on harmful images
     * Zero-shot classification capability
     * Detects context-dependent harm
     * Cross-modal verification

7.3 GRAPH NEURAL NETWORK (GNN) - USER TRUST & POST RISK

A) GRAPH CONSTRUCTION (graph_builder.py)
   - Type: Graph Data Structure
   - Library: Custom implementation
   
   - Graph Structure:
     * Nodes:
       - Users: Community members
       - Posts: User-generated content
     
     * Edges:
       - user → post (posted)
       - user → post (liked)
       - user → post (commented)
       - user → post (reported)
       - user ↔ user (follows, optional)
     
     * Edge Weights:
       - posted: 1.0 (strong connection)
       - liked: 0.5 (moderate)
       - commented: 0.7 (moderate-strong)
       - reported: 1.5 (very strong signal)
   
   - User Node Features (7 dimensions):
     1. posts_count / 100.0 (normalized)
     2. harmful_posts / 10.0
     3. reports_received / 10.0
     4. reports_made / 10.0
     5. account_age_days / 365.0
     6. trust_ratio (computed)
     7. is_user flag (1.0)
   
   - Post Node Features (7 dimensions):
     1. likes_count / 50.0
     2. comments_count / 20.0
     3. reports_count / 5.0
     4. is_flagged (1.0 or 0.0)
     5. age_days / 30.0
     6. engagement_score / 100.0
     7. is_post flag (0.0)
   
   - Trust Ratio Calculation:
     trust_ratio = 1.0 - (harmful_posts / posts_count) 
                       - min(reports_received × 0.1, 0.5)
     Clamped to [0, 1]

B) GNN ARCHITECTURE (gnn_model.py)
   - Type: Deep Learning - Graph Neural Network
   - Library: PyTorch Geometric
   - Architecture: GraphSAGE or GAT (configurable)
   
   - GraphSAGE Architecture:
     * Layer 1: Input (7 dims) → Hidden (64 dims)
       - Aggregation: Mean of neighbor features
       - Transformation: Linear(7, 64)
       - Activation: ReLU
       - Dropout: 0.3
     
     * Layer 2: Hidden (64 dims) → Output (32 dims)
       - Aggregation: Mean of neighbor features
       - Transformation: Linear(64, 32)
       - Output: Node embeddings
     
     * Trust Predictor:
       - Input: Node embedding (32 dims)
       - Layer 1: Linear(32, 16) + ReLU + Dropout(0.2)
       - Layer 2: Linear(16, 1) + Sigmoid
       - Output: Trust score (0-1)
   
   - GAT Architecture (Alternative):
     * Layer 1: Input (7 dims) → Hidden (64 dims)
       - 4 attention heads (concat)
       - Attention mechanism learns neighbor importance
       - Output: 64 × 4 = 256 dims
     
     * Layer 2: Hidden (256 dims) → Output (32 dims)
       - 1 attention head
       - Output: 32 dims
     
     * Trust Predictor: Same as GraphSAGE
   
   - Training (Semi-Supervised):
     * Loss Function:
       a) Embedding regularization: 0.01 × ||embeddings||
       b) Trust-feature correlation:
          - Users with high reports_received → low trust
          - MSE loss: (trust, 1 - reports_feature)
       c) Total loss = emb_loss + trust_loss
     
     * Optimizer: Adam (lr=0.01)
     * Epochs: 50-100
     * Batch: Full graph (small community)
   
   - Why GraphSAGE:
     * Scalable to large graphs
     * Inductive learning (new users/posts)
     * Fast inference
     * Captures multi-hop relationships
   
   - Why GAT (Alternative):
     * Attention mechanism
     * Learns importance of different neighbors
     * More expressive
     * Better for complex patterns
     * Slower than GraphSAGE

C) TRUST & RISK SCORING
   - User Trust Score:
     * Input: User node embedding (32 dims)
     * Output: Trust score (0-1, higher = more trustworthy)
     * Interpretation:
       - 0.8-1.0: Highly trusted user
       - 0.5-0.8: Moderately trusted
       - 0.0-0.5: Low trust, potential risk
   
   - Post Risk Score:
     * Input: Post node embedding (32 dims)
     * Output: Trust score (0-1)
     * Convert to risk: risk = 1 - trust
     * Interpretation:
       - 0.0-0.3: Low risk
       - 0.3-0.6: Moderate risk
       - 0.6-1.0: High risk

7.4 HYBRID MODERATION DECISION (moderation_service.py)
   - Algorithm: Weighted Multi-Signal Fusion
   
   - Final Risk Score Formula:
     final_risk = α × text_toxicity 
                + β × image_risk 
                + γ × (1 - user_trust)
   
   - Default Weights:
     * α = 0.4 (40% text)
     * β = 0.3 (30% image)
     * γ = 0.3 (30% user behavior)
   
   - Decision Thresholds:
     * final_risk ≥ 0.7 → BLOCK (immediate removal)
     * 0.5 ≤ final_risk < 0.7 → REVIEW (manual moderation)
     * 0.3 ≤ final_risk < 0.5 → FLAG (monitor)
     * final_risk < 0.3 → APPROVE (safe)
   
   - Why This Approach:
     * Multi-modal: Considers text, image, and behavior
     * Configurable: Adjust weights based on priorities
     * Contextual: User history influences decision
     * Balanced: No single signal dominates
     * Explainable: Clear breakdown of scores

7.5 WEIGHT TUNING STRATEGIES
   - Text-Focused (α=0.7, β=0.2, γ=0.1):
     * Use when: Text is primary content
     * Example: Comment sections, forums
   
   - Image-Focused (α=0.2, β=0.7, γ=0.1):
     * Use when: Visual content is primary
     * Example: Photo sharing, memes
   
   - Trust-Focused (α=0.2, β=0.2, γ=0.6):
     * Use when: User behavior is critical
     * Example: High-risk communities, fraud prevention
   
   - Balanced (α=0.33, β=0.33, γ=0.34):
     * Use when: All signals equally important
     * Example: General social media
